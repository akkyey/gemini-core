# `/refactor` ワークフロー (Refactoring Blueprint formulation)

このワークフローは、システムの**全体最適（冗長排除、アンチパターンの払拭、モジュール・インターフェースの再設計）**に専念するための分析・構想モードです。
実装へ直行せず、`/deepen` フェーズでの深い議論の「土台（Blueprint）」を作成することを最終目的（報酬）とします。

## ⚠️ 【最重要制約】エージェントの行動規範
- **例外なきコード変更の禁止**: どんなに軽微な修正（警告の解消、未使用変数の削除など）であっても、このモード実行中は**一切のファイル編集 (`replace_file_content`, `write_to_file` 等) を厳禁**とします。「軽微だから直しておこう」という独自判断・拡大解釈はフロー違反（減点対象）です。すべて記録（リストアップ）に留めてください。
- **機能追加の絶対禁止**: 新しいビジネスロジックの追加を含んではなりません。
- **最終報酬**: 「完璧なリファクタリング・ブループリント（後述）」を出力し、次のアクション（`/deepen`）へ接続すること。

## 実行ステップ

### 1. 「全体最適化モード」の開始宣言
直ちにコード編集ツールから手を放し、以下の宣言を行います。
> 🏗️ **リファクタリング構想モードを開始します。これよりいかなるコード修正・機能追加もいっさい封印し、「Blueprint（全体最適化の青写真）」の策定に専念します。**

### 2. ベースライン監査 (現状分析と不変条件の特定)
提案の安全性を担保するため、以下の分析を行います。
1. **`quality_guard` の活用**: 現状の技術的負債、静的解析のスコア、Lint違反等を把握する。
2. **`integrity_verifier` 思考の適用**: 対象モジュールが「絶対に守らなければならない外部インターフェース（入出力の型、ファイル仕様、永続化されるデータ構造等）」を特定し、**変更してはならない不変条件（ベースライン）**として明文化する。

### 3. 全体最適の構想 (`code_refactorer` 思考の展開)
局所的なリファクタリングではなく、大局的な視点で以下の改善案をプロットします。
- **アンチパターンの排除**: システムに存在する「車輪の再発明」や「過度な密結合」の特定。
- **フレームワークの最大活用**: 既存の独自実装を、`Pydantic`, `pathlib`, `collections`, `itertools` 等の標準・デファクトライブラリで代替・簡略化できないか検討する。
- **モジュール・インターフェース構成の改善案**: どのようにファイルを分割・集約し、関数シグネチャを整理すれば保守性が最大化されるかを設計する。

### 4. 【必須】ブループリントの生成と永続化
上記の調査・推論結果を統合し、**Markdown形式のBlueprint（設計書）**として指定パスへ出力（物理保存）します。

- **出力先ルール**: `docs/refactor/YYYY-MM-DD_refactor_blueprint.md` （プロジェクト内の `docs/` 配下。必要に応じてサフィックス付与）
  - **ディレクトリ作成要件**:
    1. 出力前にプロジェクトルートの `docs/` が存在するか確認する。
    2. 存在しない場合、現在のプロジェクト名を推論し、実体である `~/dev/gemini-docs/<プロジェクト名>/` および `~/dev/gemini-docs/<プロジェクト名>/refactor/` を作成した上で、プロジェクトルートにシンボリックリンク (`ln -s ~/dev/gemini-docs/<プロジェクト名> docs`) を張る。
    3. ディレクトリ作成やリンク作成に失敗した場合、独断で他の場所へ出力せず、**「docsディレクトリの実体やリンクの構築に失敗しました。対応を指示してください」とユーザーにエラーを報告し、判断を仰ぐ**こと。

**【Blueprint 必須構成要素】**
1. **リファクタリングの目的とスコープ**: 今回の最適化の対象範囲と狙い。
2. **特定されたアンチパターンと技術的負債**: 例外なくすべてリストアップすること（自身で修正しない）。
3. **不変条件（絶対守るべき入出力）**: 実装完了後の `integrity_verifier` フェーズで完全一致を証明すべき命題。
4. **アーキテクチャ変更案**:
   - モジュール・ファイル構成のビフォー・アフター
   - インターフェース（入出力の型や引数）のビフォー・アフター
   - 推奨する外部ライブラリへの置換および導入案
5. **予想されるリスクと移行措置**: 「ここを変えると他への影響が大きい・連鎖的に壊れる」といったクリティカルな分析予想。

### 5. 次フェーズへの接続（`/deepen`への誘導）
Blueprintの物理出力が完了したら、生成したファイルのパスを明示し、ユーザーに対し以下の通り問いかけます。

> 📄 **リファクタリングのBlueprintを `docs/refactor/...` に出力しました。この全体最適化案について、`/deepen`（設計深化モード）で議論と合意形成を開始しますか？**

ユーザーが合意した場合、このBlueprintを入力として `/deepen` モードへ遷移します。
